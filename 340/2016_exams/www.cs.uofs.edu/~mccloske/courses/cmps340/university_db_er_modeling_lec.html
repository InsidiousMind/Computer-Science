Development of University database via ER modeling

Adapting the example introduced in Section 1.2 of Fundamentals of Database Systems (6th edition) by Elmasri and Navathe, let's use ER modeling (as described in Chapter 7 (of the 6th edition of that book, but Chapter 3 in previous editions) to design a (very modest) database for a university.

Suppose that requirements analysis has resulted in the following (informal) description of the (entity types that exist in the) UNIVERSITY miniworld:

Each student

  • has a name (composed of a first name, last name, and middle initial)
  • has a unique ID number (a Royal Number at the U of S)
  • is in some class (either freshman, sophomore, junior, or senior)
  • has one or more majors
Each course
  • has a name (e.g., "File Processing")
  • has a unique identifier (e.g., "CMPS 340")
  • is worth some number of credit hours
  • is offered by some academic department
  • has zero or more prerequisite courses
Each course section
  • was (or "is" or "will be") an offering of some course
  • has a number (e.g., 2)
  • was offered during some semester (e.g., fall)
  • was offered during some year (e.g., 2007)
  • had some faculty member who was its instructor
  • was taken by some set of students, each of whom earned some grade
Note that a course section is uniquely identified by the first four attributes listed.

Using the above informal description, we obtain as a preliminary design the following schema (which would be better displayed as an ER diagram):

  • STUDENT(Name(First,Last,MI), Number, Class, {Major})
  • COURSE(Name, ID, Credits, Dept, {Prerequisite})
  • SECTION(Course, Number, Semester(Season, Year), Instructor, {GradeReports(Student,Grade)})

A relationship captures the idea of there being associations between (usually two, but sometimes more) entities.

If some attribute refers to an entity, it indicates the existence of such a relationship. In this example, the attributes referring to entities are Course (in SECTION), Student (in SECTION), and Prerequisite (in COURSE).

We refine the entities-only ER diagram to obtain one that depicts the relationships (by replacing attributes that refer to entities by relationships). Notion of relationships having cardinality and participation constraints.

Now use procedure spelled out in Chapter 9 (of 6th editon, but Chapter 7 in previous editions) to translate the ER diagram into a relational schema.
